The λ-Calculus: A Theory of Functions

What is Lambda Calculus?

The λ-Calculus is a model of computation. Its origins lie in research into the foundations of mathematics itself with the works of Alonzo Church in the 1930s. It allows us to compute using mathematical functions.

At its heart, a function is a mathematical object that maps an input to an output. There are many different ways to represent this operation.

f(x) = x²
y = x²
f : x ↦ x²

In the λ-Calculus we use the greek letter λ. So taking the examples above we can write a notation that does the same thing, maps an input to an output — and we do it like this.

λx.M

We take a function denoted by λ with an arbitrary input x and map that to an arbitrary output M. M is just a placeholder, the beauty of abstraction is that the symbol used doesn't matter. These variables could be whatever you want; it doesn't change anything, which ties in to...

Alpha-Equivalence

Functions are α-equivalent if they only differ in the names of their bound variables. Changing the variable name does not change the function's behavior.

λx.x + 1
λy.y + 1
λz.z + 1

All three expressions above are α-equivalent, they describe the same function, just with different variable names. The process of renaming a bound variable is called α-conversion.

Lambda Abstraction

The proper term for the lambda notation is Lambda Abstraction. It binds a variable and creates a function from an expression.

λx.x + 1

Applying this function to an input is called function application. Substituting 3 for x gives us 3 + 1 = 4. This substitution process is known as beta reduction, often written as:

(λx.x + 1) 3 → β 4

To generalize this if we have a lambda abstraction λx.M applied to an input N, we can perform a beta reduction by going through M and substituting out variable N for the variable x.

(λx.M) N → β → M[N/x]

Higher-Order Functions

Lambda Calculus is powerful because functions can take other functions as inputs and return functions as outputs. These are called higher-order functions.

λx.λy.x + y
(λx.λy.x + y) 1 → β λy.1 + y

λy.1 + y
(λy.1 + y) 2 → β λy.1 + 2

3

Each time you apply a function, you remove one λ, so you're substituting the argument into the body. Notice at the end we just keep the result after every function has been applied.

Sequentially applying multiple inputs to functions that return other functions is called currying. It allows us to handle multiple arguments in a functional way.

Computing Everything

We have three things: variables, abstraction, and application. No numbers. No booleans. No loops. The question is, can that really be enough to compute anything?

Let's find out by building the things we take for granted from scratch.

Church Numerals

There are no numbers in Lambda Calculus. So how do we represent them? Church's insight was to encode a number as the act of repetition, the number n is represented as "apply a function n times".

0 = λf.λx.x
1 = λf.λx.fx
2 = λf.λx.f(fx)
3 = λf.λx.f(f(fx))

Zero means "apply f zero times". One means "apply f once". Two means "apply f twice". The number is the repetition, not a symbol standing in for a quantity, but the very act of doing something n times.

And arithmetic falls out naturally. If 2 means "do it twice" and 3 means "do it three times", then adding them just means "do it twice, then three more times". No extra machinery needed, it works directly from the encoding.

Church Booleans

There are no booleans either. But we can encode them the same way, as functions that make a choice.

TRUE = λx.λy.x
FALSE = λx.λy.y

TRUE is a function that takes two inputs and returns the first.
FALSE takes two inputs and returns the second.

There's no WAY right? Well let's see :)

λb.λx.λy.bxy
(λb.λx.λy.bxy) True M N → β λx.λy.True xy
(λx.λy.True xy) M N → β λy.True M y
(λx.λy.True M y) N → β True M N
(λx.λy.x) M N → β (λy.M) N
M

λb.λx.λy.bxy
(λb.λx.λy.bxy) False M N → β λx.λy.False xy
(λx.λy.False xy) M N → β λy.False M y
(λy.False M y) N → β False M N
(λx.λy.y) M N → β (λy.y) N
N

That function λb.λx.λy.bxy is basically an if statement. We’re not checking True or False ourselves, we’re letting them choose. True picks the first value. False picks the second. No conditionals. No branching. Just functions.

Recursion & The Y Combinator

Here's a problem: lambda functions are anonymous. They have no name, so a function cannot refer to itself — and without self-reference, how do you get recursion? How do you loop?

The answer is the Y Combinator. The trick is to pass a function to itself as an argument, so it can call its own copy without ever needing a name.

Y = λf.(λx.f(xx))(λx.f(xx))

λf.(λx.f(xx))(λx.f(xx)) Y F
→ β (λx.F(xx))(λx.F(xx))
→ β F((λx.F(xx))(λxF(xx)))
→ F(YF)

Isn't that amazing? From such a clever trick you can get a function that returns itself, applied to another function. In other words, it creates self-reference out of nothing, something lambda calculus doesn't even support directly. That’s the textbook definition of recursion: a function defined in terms of itself.

That's Everything

We now have numbers, arithmetic, booleans, conditionals, and recursion, all built from nothing but functions. Those are the building blocks every algorithm ever written ultimately reduces to.

Church proved in the 1930s that Lambda Calculus can express any computation. Around the same time, Alan Turing proved the same thing independently with a completely different model, the Turing Machine. When they compared notes, the two models turned out to be equivalent. This equivalence is known as the Church-Turing thesis: anything that can be computed, can be computed by either.
